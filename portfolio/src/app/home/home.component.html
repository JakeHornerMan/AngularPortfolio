High Impact Contributions

Health Check Separation

The health check system was an endpoint that allowed internal and external customers to check the status of the ICCP application. This functionality was split into endpoints, shallow health returning a simple “status: UP” or “status: DOWN”, and deep health returning an in-depth analysis of all components relating to the ICCP application. These 2 endpoints would gather this information from cached values stored and logged by a background service every 30 seconds.

I was tasked with separating these two endpoints functionality, and making them have independent run times for caching and logging, 5 seconds for shallow, and 5 minutes for deep. Also, I was tasked with having the shallow endpoint return the cached value, while the deep endpoint would run the health check service independently and return the current status values.

I focused on the separation of the Heath check service into two by creating a DeepHealthService class and a ShallowHealthService class, also creating independent flags and variables to support this separation.

I used multithreading to create another scheduling task, that will be running DeepHealthService. I also created another cache store and logging functionality, for this value to be stored every 5 minutes. The previous thread handling the old implementation of health check service, was then repurposed to run the ShallowHealthService.

Finally, I pointed the deep endpoint that accessed the cached value, to now do an independent call to my DeepHealthService, and return this value to the end user.

Hashi Migration

There are many different applications that come together to make up the ICCP Platform. When each application contacts the other they have a passphrase type validation to verify that another application is from the MasterCard suite of applications. My Team, Team Vision was in charge of migrating this system to a new solution HashiCorp Vault. We were in charge of migrating CommercialAPI, PurchaseControlUI, CommercialApi Batch, and CommercialApi Report.

Mastercard has many different environments where applications are deployed and tested, Stage 1, Stage 2, Performance testing, MTF, MTF2. On top of these servers, we also need to migrate the servers based in the United States and India. We were provided with a ‘Cookbook solution’ by an alternative team on basic settings and procedures we should be doing to migrate servers.

As a Team, we had to develop a set of Instructions and Validations that needed to be carried out by members of Mastercard (BizOps) who have in-depth access to all server and Jenkins configurations. We created Pull Requests for each environment, that merged the new Hashi configurations on each of these specific servers. When these instructions were finalised, we needed to create a CRQ session with BizOps to carry out our steps and validations.

A major part I played in the Hashi Migration CRQ sessions with BizOps, was for the 8 servers in India. Mastercard cycles between an Active server set each weekend, 2 sets for 4 application. Due to our ICCP platform being a business solution, the best time to do this migration was on the weekend. We did the inactive set of 4 servers on Saturday, steps for migration, and validated the application functionality. That evening an alternative team performed with a cutover of making the migrated Hashi servers active, while also making the remaining 4 servers inactive. We then proceeded with the Hashi migration of the inactive servers on Sunday.

High Impact Contributions

Application Flow Rework

There were several different loan schemes available for customers using the SBCI customer hub, where a user could select and fill out various pages to see if they were eligible for a reduced-rate loan. When a user selects a scheme it has a hard-coded set of eight pages that would need to be completed in order to apply for that loan. The introduction of the ‘Ukraine Credit Guarantee Scheme’ aimed at providing financial aid to businesses affected by the Ukraine crisis, the business needs required a new page progression to calculate if a customer was eligible for a subsidized loan.

I created a new component that allows the loan application to take in the scheme name and populate the application with the necessary components, decouple the stepper functionality from the to allow it to be populated with the passed-on information, and also allow for reused pages from other schemes to be populated with necessary legal text for it to be used in multiple scheme types.

This system has been continued, and used to support many different loan application schemes.

Responsive CSS framework

Before the release of the Customer Hub application, a major target for us was to restructure the design to fit all devices and be responsive to all screen sizes. When I joined the team I led the process with our UX designer to form a system using Flexbox CSS. Our goal was to keep the original design flow and philosophy while allowing smaller devices to be visible and intuitive.

I created a clear class-based system that allowed us to cut down on rewriting CSS code for each Angular component. We could then call clearly defined CSS classes and modifiers to allow each box of content to scale and restructure to allow the end user to access the app on any device.

Before release, we managed to restructure all of the customer-facing pages which include the login, loan application process pages, and the main menu with sub-menus. After release, we continued to bring the Flexbox CSS system to all other users including on lender and approvers.
